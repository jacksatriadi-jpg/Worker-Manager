## RINGKASAN PROYEK

Worker Manager adalah web app berbasis Node.js (Express) + Vanilla JS
untuk mengelola worker (mesin/instance) dan pengguna (user).
Tema: Dark Glassmorphism (indigo/violet), serupa Dropbox bulk-signup UI.

## STRUKTUR FILE

Worker-Manager/
├── server.js          — Express server (API backend + activation/deactivation logic)
├── worker-control.js  — GNU screen management module (screen/log/command)
├── db.js              — JSON database module (CRUD users & workers)
├── data.json          — File database (dibuat otomatis, JANGAN commit)
├── package.json       — Dependencies Node.js
├── setup.sh           — Script setup otomatis (Armbian/Debian/Ubuntu)
├── README.md          — Dokumentasi & instruksi instalasi
├── .gitignore
├── Agent.txt          — File ini
├── logs/              — Log output screen session (dibuat otomatis, JANGAN commit)
└── public/
    ├── index.html     — SPA (Single Page Application)
    └── index.css      — Semua styling (Dark Glassmorphism)

    {

## API ENDPOINTS

### Auth
POST   /api/login            — Login (body: {username, password})
POST   /api/logout           — Logout
GET    /api/me               — Get current session user

### Users (Admin only)
GET    /api/users            — Get all users
POST   /api/users            — Create user (body: {username, password, role})
DELETE /api/users/:id        — Delete user (tidak bisa hapus diri sendiri)

### Workers (Auth required)
GET    /api/workers          — Get all workers
POST   /api/workers          — Create worker (Admin) {name, workerLink, apiLink}
PATCH  /api/workers/:id/status — Update status {status: "active"|"inactive"}
DELETE /api/workers/:id      — Delete worker (Admin only)

## CATATAN TEKNIS

1. Database menggunakan JSON file (data.json), bukan SQL/NoSQL.
   Cocok untuk skala kecil. Jika perlu skala besar, migrasi ke SQLite
   (gunakan better-sqlite3) atau MongoDB.

2. Session secret ("worker-manager-secret-2024") harus diganti dengan
   variabel environment di production:
   process.env.SESSION_SECRET || 'fallback-secret'

3. bcrypt hash untuk "Bali@123" yang di-hardcode:
   $2a$10$SFubiddOHvxUtEmuZ1PC/OJr6zLUbNEHbhChQOTC7SzsPQRI9IkXC
   Jika hash tidak cocok, generate ulang dengan:
   const bcrypt = require('bcryptjs');
   console.log(bcrypt.hashSync('Bali@123', 10));
   Lalu update di db.js dan hapus data.json agar terinisialisasi ulang.

4. Port default: 8090. Bisa diubah via environment variable PORT.

5. Semua UI ada di public/index.html (vanilla JS, tidak pakai React/Vue).
   CSS ada di public/index.css (custom dark glassmorphism).

## DEPLOYMENT (ARMBIAN / LINUX)

Script setup.sh melakukan:
  1. apt-get update
  2. Install curl, git, wget, build-essential
  3. Install Node.js 18+ LTS via NodeSource (ARM32/ARM64/x86_64)
     → Jika Node.js v18.x atau lebih baru sudah ada, langkah ini dilewati
  4. Clone repo dari GitHub ke /opt/worker-manager
     → Jika sudah ada: git pull (update otomatis)
  5. npm install --omit=dev
  6. Buat .env (PORT=8090, SESSION_SECRET=random)
  7. Buat & enable systemd service 'worker-manager' (autostart)
  8. Buka port 8090 di UFW (jika UFW tersedia)

Cara menjalankan setup.sh:
  chmod +x setup.sh
  sudo bash setup.sh

Service systemd:
  sudo systemctl status worker-manager    # Cek status
  sudo systemctl restart worker-manager   # Restart
  sudo journalctl -u worker-manager -f    # Lihat log

File .env dibuat otomatis di /opt/worker-manager/.env:
  PORT=8090
  SESSION_SECRET=<random 32 hex>
  NODE_ENV=production

Untuk update aplikasi:
  sudo bash /opt/worker-manager/setup.sh  # Jalankan ulang setup.sh

## WORKER FIELDS (TERBARU)

Field worker di data.json:
  name             — Nama tampilan worker
  workerLink       — URL yang dibuka saat status clickable (Link Siap/Selesai)
  apiLink          — URL endpoint untuk polling status (GET, return JSON)
  screenName       — Nama GNU screen session yang digunakan
  gcloudCommand    — Command untuk login ke Google Cloud Shell
                     Contoh: gcloud cloud-shell ssh --configuration=protonz0002 --authorize-session
  cloudflareCommand— Command untuk menjalankan Cloudflare tunnel
  serverCommand    — Command untuk menjalankan server di dalam cloud shell
  status           — Salah satu dari status di bawah

## WORKER STATUS MACHINE

Status dan artinya:
  inactive     — Nonaktif (abu-abu) — default
  preparing    — Menyiapkan (ungu, animasi pulse+spinner) — sedang aktivasi
  ready        — Link Siap (hijau, clickable → workerLink) — API status BOOT
  processing   — Sedang Dipakai (kuning) — API status PROCESSING/BOOT
  finished     — Selesai, siap digunakan (hijau, clickable → workerLink) — API FINISH
  deactivating — Menonaktifkan (merah, animasi pulse) — API IDLE, tunggu 30s

Transisi otomatis:
  BOOT       → status 'ready'
  PROCESSING → status 'processing'
  FINISH     → status 'finished'
  IDLE       → status 'deactivating' → (30 detik) → deactivation flow → 'inactive'
  'Exception in thread' di log → langsung 'inactive'

## ALUR AKTIVASI (server.js: activateWorker)

1. Set status 'preparing'
2. Cek screen (screen -ls), buat baru jika belum ada (screen -dmS name)
3. Setup logging: screen -S name -p 0 -X logfile /path && log on
4. Kirim gcloudCommand via: screen -S name -p 0 -X stuff 'cmd\n'
5. Tunggu '@cloudshell:~$' di log (timeout 30s, retry max 3x)
6. Kirim cloudflareCommand, tunggu '@cloudshell:~$' (timeout 60s)
7. Kirim serverCommand
8. Sleep 5 detik, GET apiLink, cek status response
9. Set status (ready/processing/finished/deactivating)
10. Detach screen (screen -S name -d)
11. Mulai polling apiLink setiap 5 detik (startApiPolling)

## ALUR DEAKTIVASI (server.js: deactivateWorker)

1. Hentikan polling interval
2. Masuk ke screen yang ada
3. Setup logging
4. Kirim CTRL-C (\x03) berulang (max 5x) sampai '@cloudshell:~$'
5. Kirim 'logout\n'
6. Detach screen
7. Set status 'inactive'

## SCREEN INTERACTION (worker-control.js)

Semua command ke screen menggunakan execFile (bukan exec/shell):
  screenExists(name)         — screen -ls | cek baris berakhiran .name
  createScreen(name)         — screen -dmS name, tunggu 1 detik
  setupLogging(name, path)   — truncate log, -X logfile path, -X log on
  sendCommand(name, cmd)     — -X stuff 'cmd\n'
  sendCtrlC(name)            — -X stuff '\x03'
  detachScreen(name)         — screen -S name -d
  waitForString(logFile, str, startPos, timeoutMs)
    — baca log mulai dari startPos bytes
    — strip ANSI codes sebelum pencocokan
    — throw 'Timeout' atau 'Exception in thread'
    — return posisi byte baru jika ditemukan

Log file: /opt/worker-manager/logs/screen-<screenName>.log

  PUT    /api/workers/:id        — Edit worker (Admin) — semua field

## RECOVERY SAAT SERVER RESTART

Fungsi recoverWorkerStates() dipanggil saat server start:
  'preparing', 'deactivating' → reset ke 'inactive'
  'ready', 'processing', 'finished' → resume polling API

## CATATAN PENTING

1. GNU screen harus tersedia di sistem (apt install screen)
2. Log file otomatis dibuat di folder logs/ (tidak di-commit)
3. activePollIntervals (Map) dan workerLocks (Map) adalah in-memory;
   akan hilang saat server restart (ditangani oleh recoverWorkerStates)
4. workerLocks mencegah double-activation/deactivation
5. Frontend auto-poll setiap 5 detik via setInterval di showApp()
